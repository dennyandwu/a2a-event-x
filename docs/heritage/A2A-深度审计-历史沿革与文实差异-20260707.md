## 五、对最终方案的修订结论

1. **P0-1 看门狗(修正版)**:**不能直接加载现有 a2a-monitor 当积压告警**——live 代码实为 inter_session→Discord 同步器(见 D4 修订)。正确动作:把"a2a-monitor plist 未加载"作为 launchd 巡检缺口样本;审阅现有脚本后**改造或重写**为真正的 pending backlog watchdog(直发 Discord webhook,复用 notify-discord.sh / notify-webhook.url 基建);"launchd 服务实际加载状态"与"各 agent 事件产出速率"均纳入巡检项(见 §六)。
2. **P0 增加一项**:清理 launchd 失败项(oneshot.update-20260702 卸载、wiki-lint/cron-session-retention 退出码排查)。
3. **P1 契约基线明确**:以 06-20 对账文档 + 07-03 代码现状为 schema v2 定稿输入,9 项文档欠账全部清偿;canonical 协议文档更新为验收项。
4. **P1 git 收敛方向修正**:不是"合并两份",而是**以 live 为源 re-import**进 a2a-toolkit repo,并把 5 月后的整层组件(automation-runner/followup-runner/protocol-guard/mailbox-shadow/codex-a2a-consumer/health-check)一并纳管、各补一页式设计说明。
5. **新增决策项**:Bridge auto-post 判生死(停摆自 04-24;若 v2 用长轮询则正式废除并删 posted-seqs 机制)。
6. **文档治理落地**:canonical 文档目录定为 macmini vault `Projects/PROJ-012-Agent-Dashboard/`(历史所在地),优化方案及后续 spec 迁入;当前连接 vault 中的副本标注"镜像,以 macmini 为准",消除 D6 分裂。

## 六、5/22 → 6 月升级窗口专项排查(v1.1 补充,2026-07-07)

**背景**:6 月 OpenClaw 经历多次升级(06-11 restart、06-16 node24 迁移并重建 gateway plist、07-02 oneshot 升级失败 exit 127);疑问是 5/22 版本之后的升级是否导致大量 A2A 相关 cron/文件停摆。

**结论:cron 注册表未丢失任何启用中的任务;但升级窗口内有真实的 agent 级停摆。**

1. **5/22 → 6/1 注册表零变化**:`jobs.json.bak`(05-22)与 `jobs.json.migrated`(06-01)完全一致——108 个任务,无增删、无 enabled 翻转。这段时间没有"大量修改"。
2. **6 月上旬 cron 注册表迁移**(旧 `jobs.json` → 新存储,`runs/` 归档为 `.migrated`,时间点 06-09):109 → 51 任务。**丢失的 59 个全部是本来就 disabled 的任务**(一次性 followup、退役的 a2a:elon/satoshi/ansen/issac-poll、monitor:* 家族、cron 版 pipeline:executor 等);**0 个 enabled 任务丢失**。7 个 enabled 的 a2a cron(task-handoff / context-bridge / proactive-sync / sentiment-monitor / auto-close-completed / auto-close-overdue-ansen / compensate-dispatches)全部存活,07-07 当天全部 lastStatus=ok。
3. **真实停摆(升级窗口内)**:
   - **issac 事件产出中断约 5.9 天:06-12 00:51 → 06-17 21:46 零事件**。与 06-12 cross-surface 事故备份(a2a-send.sh / a2a_routing / pipeline-executor 均 06-12 01:49 修改)、06-13 Bridge 重启、06-16 node24 迁移(期间 launchctl 一度报 "Could not find service ai.openclaw.gateway")时间吻合;
   - **followup-runner 两段空窗**:06-06 → 06-11、06-13 → 06-14;
   - automation-runner(launchd 常驻)无中断,仅 06-19 ~ 06-27 日产出降约 25%(~95 → ~70/天)后恢复。
4. **历史"停摆"多为主动下线而非升级事故**:sop:email-check、monitor:*-health 等 cron 早在 5 月前已 disabled,职能迁移给 automation-runner 的 sop-email-check / ops-auto-alert-* topics(ansen 的 5-15 僵尸积压正是该迁移期产物);pipeline:executor 由 cron 迁至 launchd。
5. **06-27/28 FD retention 事件**(state docs 有专项文档):gateway 持有 ~7000 FD,codex 归档 transcript 未能释放已打开的 FD;该文档列出 8 个高频 agentTurn cron(含 a2a:task-handoff / context-bridge / proactive-sync 等)待迁移为 command/batch 任务——**此清单应与 v2 方案 P1 的组件纳管合并处理**。
6. **session 归档已制度化且未根治 FD 问题(v1.2 补充)**:session-fd-relief 现为每日两跑——05:10 本地归档 main/ansen/satoshi/elon 中 >7 天的 session 文件(每次 ~260-500 个,07-06 归档 main 262 个),17:00 归档 cron 中 **>0.5 天**的文件(07-06 归档 492 个,含 10 个 "event-log/a2a workflow wake" 场景 transcript)。归档控制了目录增长,但 FD 泄漏依旧:07-04/05 连续出现 exec spawn EBADF 并两度触发 gateway 重启,07-06 间歇性 a2a CLI 执行失败直接造成 issac 积压(见 §四-3)。**归档删 session 未直接造成"wake 打到旧 session"(无 ENOENT/no_session 证据),但它与 EBADF 同属 FD 事故家族:治标(归档)在跑,治本(FD 泄漏)未解。**

**对方案的影响**:升级本身不是当前积压的根因(7 个 a2a cron 都活着),但证明两点——(a) 迁移/升级窗口是 agent 级停摆高发期,看门狗必须监控**事件产出速率**(某 agent 连续 N 小时零事件即告警),而不只是 pending 数;(b) 注册表迁移会静默清理 disabled 任务,任何需要保留的任务应显式归档,不能依赖 disabled 状态存续。
